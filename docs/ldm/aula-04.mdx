## 📖 Conceito

Na branch anterior, deixamos o Supabase moldado com o nosso "plano das 6 tabelas" (Schema SQL). Mas para que nosso Ktor Server (e o CMS Desktop) consigam pedir que a classe Kotlin entenda que `streak_freezes` corresponde à uma variável do Back-end... entra a orquestração via **Data Transfer Objects** (DTOs).

Com as facilidades Kotlin Multiplatform (KMP), escrevemos esses modelos no diretório _Shared_ (`shared/.../domain/models`), permitindo que a aplicação Server, o App mobile do PDM, e o CMS de Desktop possam usar a Mesma Coleção de Classes. Se um campo mudar, a correção acontece de uma vez para os três.

## 💻 Código de Referência

### @Serializable e Mapeamento de @SerialName

As classes do projeto utilizam a tag mágica `@Serializable` provida pelo pacote `kotlinx.serialization` (que já linkamos na Aula 02 no Gradle).

Um grande desafio de engenharia cross-stack é o nome das variáveis:
* Postgres (e o JSON trafegado na web) usa tradicionalmente o modelo `snake_case`. (Ex: `created_at`).
* Kotlin defensores de código legível usam o formato `camelCase`. (Ex: `createdAt`).

Em vez de renomear os campos Kotlin de maneira agressiva, o `@SerialName` traduz de volta para a infraestrutura sem expor feiúra e sem poluir a sintaxe legível:

```kotlin title="shared/.../domain/models/Models.kt"
@Serializable
data class User(
    val id: String,
    val username: String,
    val email: String,
    @SerialName("streak_count")
    val streakCount: Int = 0,
    @SerialName("streak_freezes")
    val streakFreezes: Int = 0,
    @SerialName("last_activity_at")
    val lastActivityAt: String? = null,
    @SerialName("created_at")
    val createdAt: String? = null,
    @SerialName("updated_at")
    val updatedAt: String? = null
)
```

Nesta branch, possuímos classes isoladas que refletem completamente o banco em pastas granulares:

* `AuthDTOs.kt` → Lidando com payloads (LoginRequest, LoginResponse).
* `ContentDTOs.kt` → Coleção específica das Aulas interligadas a um Curso (Module/Trail).
* `Models.kt` → Core business logic com as classes bases (Course, User, Lesson, etc).

### Repository Pattern no KMP

E onde usamos isso? Pelo princípio SOLID de Responsabilidade Única, a lógica que pede ativamente os Cursos no banco ou que adiciona novos perfis... **não deve morar nos Routers HTTP nem nas Views dos Apps.** Ela mora em Contratos de Interface chamados "Repository".

Nas instâncias mais elementares da Aula-04 criamos as interfaces com assinaturas que retornam a estrutura KMP construída nas APIs.

```kotlin title="shared/.../domain/repository/CourseRepository.kt"
interface CourseRepository {
    suspend fun getAllCourses(): List<Course>
    suspend fun getCourseById(id: Int): Course?
    suspend fun createCourse(course: Course): Course
}
```
*A palavra-chave coroutine `suspend` no Kotlin garante que nossa requisição de rede irá para *background thread* em vez de congelar a interface (especialmente para quando chegar a vez do Compose).*
