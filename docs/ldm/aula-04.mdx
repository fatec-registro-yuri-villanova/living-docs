## 📖 Conceito

Na branch anterior, deixamos o Supabase moldado com o nosso Schema SQL de 6 tabelas (IDs baseados em `serial` inteiros, e relacionamentos definidos). Para que o Kotlin entenda que o campo SQL `streak_freezes` corresponde à uma propriedade legível no App... entra a orquestração via **Data Transfer Objects** (DTOs).

Com as facilidades do Kotlin Multiplatform (KMP), escrevemos esses modelos no diretório _Shared_ (`shared/.../domain/models`), permitindo que a aplicação Server, o App mobile do PDM, e o CMS de Desktop (PDMII) possam usar a **mesma coleção de classes**. 

## 💻 Código de Referência

### @Serializable e Mapeamento de @SerialName

As classes do projeto utilizam a anotação mágica `@Serializable` provida pelo pacote `kotlinx.serialization`.

Um grande detalhe do mapeamento de Objetos Relacionais:
* O Postgres usa o padrão `snake_case`. (Ex: `created_at`).
* Kotlin defende código no formato `camelCase`. (Ex: `createdAt`).

Em vez de quebrar a legibilidade e forçar o Kotlin a usar sublinhados nativamente nas propriedades, o `@SerialName` traduz de volta para a infraestrutura por baixo dos panos:

```kotlin title="shared/.../domain/models/Models.kt"
@Serializable
data class User(
    val id: Int,
    val username: String,
    val email: String? = null,
    val name: String? = null,
    val password: String? = null,
    @SerialName("profile_picture")
    val profilePicture: String? = null,
    @SerialName("streak_count")
    val streakCount: Int = 0,
    @SerialName("last_activity_at")
    val lastActivityAt: String? = null,
    @SerialName("streak_freezes")
    val streakFreezes: Int = 0,
    @SerialName("longest_streak")
    val longestStreak: Int = 0,
    val role: String = "user",
    @SerialName("created_at")
    val createdAt: String? = null
)
```

:::tip Tipos Nativos Primitivos
Ao ler nosso `initial_schema.sql`, notamos que as Primary Keys (`id`) são `serial`, um auto-incremento numérico. Portanto, no Kotlin, nossos campos de `id` são todos mapeados para a tipagem primária `Int`. Identificadores relacionais (foreign keys) também acompanham essa formatação via `Int?`.
:::

Nesta branch, possuímos classes isoladas que refletem o banco em pastas granulares:

* `AuthDTOs.kt` → Lidando com payloads (LoginRequest, LoginResponse).
* `ContentDTOs.kt` → Coleção específica das Aulas interligadas a um Curso (Module/Trail).
* `Models.kt` → Core business logic com as classes bases (Course, User, Lesson, etc).

### Repository Pattern no KMP

Pelo princípio SOLID de Responsabilidade Única, a lógica de pedir ativamente os Cursos no banco ou adicionar novos perfis... **não deve morar nos Routers HTTP nem nas Views dos Apps.** Ela mora em Contratos de Interface chamados "Repository".

Nas instâncias mais elementares da Aula-04 criamos as interfaces com assinaturas que retornam a estrutura base das Entidades construídas:

```kotlin title="shared/.../domain/repository/UserRepository.kt"
interface UserRepository {
    suspend fun login(request: LoginRequest): Result<AuthResponse>
    suspend fun register(request: RegisterRequest): Result<AuthResponse>
    suspend fun getUserProfile(userId: Int): Result<User>
    suspend fun updateStreak(userId: Int): Result<User>
}
```
*A palavra-chave coroutine `suspend` no Kotlin garante que nossa requisição de rede irá para *background thread* em vez de congelar a interface (especialmente para quando chegar a vez do Compose no Desktop ou Mobile).*
