# Aula 02: Supabase Client Compartilhado

:::info 🔀 Branch da Aula
Esta aula refere-se à branch: **`aula-02`**
```bash
git checkout aula-02
```
:::

## 📖 Conceito

Na aula anterior, montamos a estrutura KMP com um servidor Ktor puro. Agora vamos dar o próximo passo: **conectar o projeto ao Supabase** — o banco de dados que alimentará nosso ecossistema MergeSkills.

O segredo está no módulo `shared/`. Como ele é compilado para múltiplas plataformas (JVM, Android, Desktop), qualquer código escrito lá fica disponível automaticamente para todos os módulos. Isso significa que vamos criar **uma única configuração** do Supabase e reutilizá-la em:

- 🖥️ **Server** (Ktor) — para endpoints da API
- 💻 **Desktop** (Compose for Desktop) — para o CMS administrativo
- 📱 **Android** (futuro) — para o app mobile

### O que muda nesta branch (vs aula-01)

| Arquivo | O que foi adicionado | Módulo |
|---------|---------------------|--------|
| `SupabaseClient.kt` | Factory function para criar o client | `shared/` |
| `Constants.kt` | Porta padrão do server | `shared/` |
| `shared/build.gradle.kts` | Dependências Supabase SDK | `shared/` |
| `server/build.gradle.kts` | Dependências Supabase + Database | `server/` |
| `Application.kt` | Inicialização do SupabaseClient | `server/` |

### Supabase SDK — O que são os Plugins?

O Supabase SDK para Kotlin é modular. Cada funcionalidade é um **plugin** que você instala separadamente:

| Plugin | Função | Quando usar |
|--------|--------|-------------|
| `Postgrest` | Acesso ao banco de dados (CRUD) | Sempre — é o core |
| `Auth` | Autenticação de usuários | Login, Register, JWT |
| `Storage` | Upload/download de arquivos | Fotos de perfil, imagens |

## 💻 Código de Referência

### SupabaseClient.kt — Factory Function no Shared

A função `createAppSupabaseClient` vive no `commonMain`, o que significa que **qualquer módulo** pode chamá-la com suas próprias credenciais:

```kotlin title="shared/src/commonMain/kotlin/.../SupabaseClient.kt"
fun createAppSupabaseClient(
    supabaseUrl: String,
    supabaseKey: String
): SupabaseClient {
    return createSupabaseClient(
        supabaseUrl = supabaseUrl,
        supabaseKey = supabaseKey
    ) {
        install(Postgrest)   // Acesso ao banco (queries, inserts, etc.)
        install(Auth)        // Autenticação (login, register, JWT)
        install(Storage)     // Armazenamento de arquivos (fotos, docs)

        defaultSerializer = KotlinXSerializer(Json {
            ignoreUnknownKeys = true  // Ignora campos extra do JSON
            prettyPrint = true        // JSON formatado nos logs
            isLenient = true          // Aceita JSON "relaxado"
        })
    }
}
```

:::tip Factory Pattern
Uma **factory function** é uma função que cria e retorna um objeto pronto para uso. Em vez de chamar `createSupabaseClient(...)` espalhado pelo código, centralizamos a configuração em um lugar só. Se amanhã quisermos adicionar um plugin novo, alteramos apenas este arquivo.
:::

### Application.kt — Server consome o Shared

O servidor lê as credenciais via `System.getProperty()` (JVM args definidos no `build.gradle.kts`) e usa a factory do shared:

```kotlin title="server/src/main/kotlin/.../Application.kt"
fun Application.module() {
    // Credenciais vêm do build.gradle.kts → local.properties
    val supabaseUrl = System.getProperty("SUPABASE_URL")
    val supabaseKey = System.getProperty("SUPABASE_KEY")

    var supabase: SupabaseClient? = null

    if (supabaseUrl != null && supabaseKey != null) {
        // Reutiliza a factory do módulo shared!
        supabase = createAppSupabaseClient(supabaseUrl, supabaseKey)
    }

    install(ContentNegotiation) {
        json(Json {
            prettyPrint = true
            isLenient = true
            ignoreUnknownKeys = true
        })
    }

    routing {
        get("/") {
            call.respondText("API LDDM/PDM (mergeskillskmp). Tente acessar /api/health")
        }

        get("/api/health") {
            call.respond(
                HealthResponse(
                    status = "Status UP",
                    timestamp = System.currentTimeMillis()
                )
            )
        }
    }
}
```

### Dependências — O que foi adicionado no Gradle

No `shared/build.gradle.kts`, três novas dependências com `api()` (visíveis para quem depende do shared):

```kotlin title="shared/build.gradle.kts (diff vs aula-01)"
commonMain.dependencies {
    // ... (existentes da aula-01)
    api(libs.supabase.postgrest)  // NOVO: Database
    api(libs.supabase.gotrue)     // NOVO: Auth
    api(libs.supabase.storage)    // NOVO: Storage
}
```

:::tip `api()` vs `implementation()`
- **`api()`**: A dependência fica visível para quem depende deste módulo (o server "enxerga" o Supabase automaticamente).
- **`implementation()`**: A dependência fica privada — só o shared a conhece.

Como queremos que o server use o SupabaseClient diretamente, usamos `api()`.
:::

Modifique o endpoint `/api/health` para retornar se o Supabase está conectado ou não:

get("/api/health") {
    call.respond(HealthResponse(

        timestamp = System.currentTimeMillis()
    ))

### Primeira query real

Crie um endpoint `GET /api/courses` que use o Supabase para buscar dados da tabela `courses`:

```kotlin
get("/api/courses") {
    val courses = supabase?.postgrest?.from("courses")?.select()
    call.respondText(courses?.data ?: "[]")
}
```

:::tip Cuidado ao rodar o Servidor Local!
Se você clicar no "Play" verde direto no arquivo `Application.kt` via Android Studio/IntelliJ, a sua IDE **não lerá** o arquivo `local.properties` e o Ktor subirá sem as credenciais do Supabase (retornando `[]` vazio na API).

**Sempre rode o servidor via Gradle:**  
No terminal digite `./gradlew server:run` (ou rode a task Ktor diretamente pela aba do Gradle no canto direito). Assim, o `build.gradle.kts` irá injetar os `systemProperties` e a API conseguirá pegar as listas de Cursos do Supabase!
:::

Se a tabela ainda não existir, o Supabase retornará um erro 404 — e tudo bem, na próxima aula criaremos o schema!
